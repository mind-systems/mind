Weighted Continuous Selection Pool — System Overview

The Weighted Continuous Selection Pool (WCSP) is the core economic model of the Bio-Proof-of-Brain mining system. It replaces discrete epoch-based reward allocation with a continuous, state-driven participation pool where users are included or excluded based on real-time physiological eligibility and contribute proportionally to reward distribution based on computed biological hashrate.

The system is designed around the principle that biological contribution is not a discrete event but a continuous state, and therefore reward participation must reflect continuous dynamics rather than time-sliced computation windows.

At any moment in time, the system maintains an active pool of eligible users. Eligibility is determined through a multi-layer threshold system applied across independent biometric modules such as respiratory regulation, EEG coherence, cardiovascular recovery, and motion stability. Each module produces a normalized scalar contribution, which is then evaluated against configurable thresholds defined by the administrative governance layer.

A user is considered eligible for participation only if their aggregated module-level state satisfies the system-wide eligibility function. Eligibility is not binary in its economic effect; it produces a continuous weighting factor that influences the user's effective participation strength inside the pool.

Once a user is admitted into the pool, their economic influence is determined by a unified hashrate scalar. This scalar is computed as a weighted combination of all active biometric modules, where each module contributes according to its configured weight, signal confidence, and user-specific physiological profile modifiers derived from long-term adaptation modeling.

The resulting hashrate does not represent absolute biological performance but rather normalized contribution potential within the system’s economic model. This hashrate becomes the primary unit of influence inside the continuous selection pool.

Reward distribution occurs proportionally across all eligible users in the pool. The system does not rely on random selection or probabilistic lottery mechanisms. Instead, reward allocation is deterministic and directly proportional to each user's effective weighted hashrate relative to the total pool hashrate.

This architecture allows the system to operate continuously without requiring fixed temporal epochs as a primary economic construct. Instead, time is used only as a backend batching mechanism for ledger consistency and blockchain settlement, while the economic model remains state-driven and continuously updated.

The WCSP model is designed to support dynamic module expansion. New biometric inputs can be added to the system without modifying core economic logic, as all modules conform to a standardized interface that produces a normalized scalar output. This ensures long-term extensibility of the protocol across future sensor integrations.

Administrative governance plays a central role in system configuration. All module weights, threshold parameters, and reward policy constraints are fully configurable through a role-based access control system. This allows the system to adapt to different physiological domains, research updates, and economic balancing requirements without requiring protocol-level redeployment.

The Weighted Continuous Selection Pool therefore represents a shift from time-based mining systems to state-based biological participation systems, where economic influence emerges from real-time physiological regulation quality rather than discrete computational or temporal events.

## Core Concept Layer — Pool Model (Technical Specification)

The Core Concept Layer defines the fundamental economic abstraction of the entire system: the Weighted Continuous Selection Pool. This layer is responsible for maintaining the real-time representation of all users participating in the reward economy and determining, at any given moment, who is economically active and how strongly each participant contributes to the global reward distribution.

The system is designed around a continuous state model rather than discrete temporal events. This means that the pool does not “tick” in a way that defines reward distribution intervals as primary units of logic. Instead, the system maintains an always-updating snapshot of participant states, and all economic computation is derived from this evolving state. Time exists only as a supporting dimension for batching, persistence, and settlement, but never as a core determinant of reward logic.

At the center of this layer is the concept of a pool membership state. Every user in the system is represented as a dynamic entity that can either be inside or outside the active reward pool at any given moment. Membership is not determined by identity or subscription status, but by physiological and behavioral validity as computed by upstream systems such as signal reduction, quality evaluation, fraud detection, and module-level scoring. A user enters the pool only when their aggregated biological contribution satisfies the eligibility function defined by the system configuration.

Once a user is inside the pool, their participation is not binary. Membership defines only inclusion; economic influence is defined separately. Each participant has a continuously computed effective weight that determines their proportional influence over reward distribution. This weight is derived from their current hashrate, eligibility strength, trust modifiers, and confidence values produced by quality evaluation systems. The resulting value is always a non-negative scalar that represents relative economic significance within the active pool.

The pool itself is therefore not a list in the traditional sense, but a continuously evolving weighted distribution space. At any given moment, the sum of all effective weights across all eligible users forms the normalization denominator for reward allocation. Individual user reward share is computed as the ratio of their effective weight to the total weight of the pool. This ensures that reward distribution remains proportional, deterministic, and independent of any random selection mechanism.

The system intentionally avoids probabilistic lottery behavior. Although biological signals may be noisy and user states may fluctuate rapidly, the economic model is strictly deterministic. Any apparent randomness in outcomes emerges solely from continuous variation in input signals rather than from explicit stochastic selection logic.

The pool model is also designed to support high-frequency state updates without requiring full recomputation of global state. Each user update affects only local contributions to the pool, and the system maintains incremental aggregation of total weight. This allows the pool to scale horizontally without collapsing under global recomputation costs.

Membership transitions, meaning the process of entering or leaving the pool, are governed by eligibility evaluation outputs. However, these transitions are not instantaneous binary flips in economic effect. Instead, the system introduces a controlled transition behavior where changes in eligibility propagate through weighted influence adjustments rather than abrupt inclusion or exclusion. This prevents oscillation artifacts where users repeatedly enter and exit the pool due to borderline physiological states.

The Core Concept Layer therefore defines the minimal invariant of the entire protocol: at any point in time, there exists a continuously updated set of users with associated effective weights, and all economic reward logic is a direct function of the normalized distribution of those weights. Everything else in the system, including module scoring, fraud detection, coaching adaptation, and reputation modeling, ultimately influences this layer either by modifying eligibility or by adjusting the effective weight computation, but never by altering the structural definition of the pool itself.

This layer is intentionally kept minimal and mathematically stable because it serves as the foundation upon which all higher-level economic and behavioral systems are constructed.

## Module System — Generic Signal to Hashrate Conversion (Technical Specification)

The Module System defines the standardized interface through which all biological signals are translated into a unified economic representation. Its purpose is to ensure that any present or future biometric input, regardless of sensor type, signal structure, or physiological domain, can be integrated into the system without requiring changes to the core economic logic of the protocol.

The fundamental design principle of this layer is strict normalization. Every module, whether it processes EEG activity, cardiac rhythms, respiratory patterns, motion dynamics, or any future sensor modality, must produce a single scalar output that is semantically equivalent across the entire system. This scalar is referred to as module hashrate contribution, and it represents the economic value of a specific physiological domain at a specific point in time.

Each module operates as an independent processing unit that consumes raw or partially reduced biometric streams and transforms them into structured internal features. These features are then evaluated through a module-specific scoring function that captures the quality, stability, and meaningfulness of the underlying biological signal. The scoring function is not universal across modules because each physiological domain has fundamentally different signal characteristics, noise profiles, and temporal dynamics. However, despite this internal diversity, all modules must converge to a shared output contract.

The output of every module is a normalized scalar value representing contribution strength. This value is always bounded and calibrated against a user-specific physiological baseline model, ensuring that absolute values are never used directly for economic computation. Instead, all outputs are expressed as relative deviations from long-term behavioral and biological norms of the user. This prevents systemic bias toward inherently high-amplitude signal types and ensures fairness across heterogeneous sensor modalities.

In addition to the primary hashrate contribution value, each module produces a confidence signal that reflects the reliability of the underlying measurement at the time of computation. Confidence is influenced by factors such as artifact detection, signal dropout, sensor instability, and internal coherence of the measured biological state. This confidence value does not directly represent biological quality but rather measurement trustworthiness, and it plays a critical role in downstream weighting within the aggregation layer.

Each module also defines an internal mapping function that converts raw physiological inputs into its normalized scoring domain. This mapping function is configurable but must remain deterministic and replayable. The system explicitly forbids any non-deterministic transformations within modules, as reproducibility of historical states is a core requirement for auditability, fraud detection, and reward settlement consistency.

Although modules are structurally independent, they are not isolated in effect. All module outputs are ultimately consumed by the unified hashrate aggregation engine, where they are combined into a single scalar representation of user economic contribution. This means that module design must always consider cross-module interaction indirectly, particularly in terms of scaling behavior and weight calibration.

Module weights are not hardcoded inside the module definitions themselves. Instead, they are external configuration parameters controlled by the administrative governance layer. This separation ensures that the same module implementation can be reused across different system configurations, research phases, or sensor deployments without requiring code-level modification. It also allows dynamic rebalancing of the economic importance of different physiological signals as the system evolves.

The Module System is explicitly designed to be extensible. Adding a new biometric modality does not require modification of the aggregation logic, eligibility engine, or reward system. A new module is simply introduced as a conforming implementation of the standard interface, producing a normalized scalar output and confidence value. Once registered in the system configuration, it becomes immediately part of the global hashrate computation pipeline.

Despite this flexibility, all modules are required to adhere to strict constraints regarding determinism, normalization, and bounded output behavior. Any module that violates these constraints is considered invalid for economic computation and is excluded from reward influence. This ensures that the system remains stable even as new and potentially complex biometric inputs are introduced over time.

In essence, the Module System serves as the translation layer between heterogeneous biological reality and a unified economic representation. It abstracts away sensor-specific complexity while preserving meaningful physiological variation in a controlled, normalized form that can be safely used in continuous reward computation.

## Threshold & Eligibility Engine (Technical Specification)

The Threshold & Eligibility Engine defines the gatekeeping layer between raw biological contribution and economic participation in the Weighted Continuous Selection Pool. Its responsibility is to determine whether a user’s current physiological state is sufficient to allow participation in reward distribution and, if so, with what degree of influence. This layer does not compute hashrate itself and does not modify module outputs; it only evaluates whether the aggregated result of upstream computation is valid for inclusion in the economic pool and how strongly that inclusion should be expressed.

The system operates on the principle that biological signals must reach a minimum level of structural coherence before they are considered economically meaningful. This is necessary because raw physiological data inherently contains noise, instability phases, and transitional states that do not represent intentional or stable user behavior. The Eligibility Engine therefore functions as a filter that separates economically relevant physiological regulation from incidental or unstructured signal activity.

At its core, the Eligibility Engine consumes the output of all active modules after they have been normalized into their scalar contribution values. It does not access raw signal streams directly. Instead, it evaluates a structured representation of user state that includes per-module hashrate contributions, confidence values, and derived stability indicators produced by upstream systems. This ensures that eligibility decisions are made on already-processed, deterministic data rather than raw, noisy inputs.

Eligibility is not modeled as a simple binary condition in its full form. Instead, the engine computes a continuous eligibility strength value that represents the degree to which a user qualifies for participation in the reward pool. This value exists in a bounded range between zero and one, where zero represents complete exclusion from the pool and one represents full participation under ideal conditions. Values between these extremes represent partial or transitional participation states where the user is included in the pool but with reduced economic influence.

The computation of eligibility strength is based on threshold functions defined per module and per system configuration. Each module contributes an independent eligibility score that reflects whether its current state exceeds a configured minimum viability threshold. These thresholds are not fixed constants embedded in the system logic but are externally configurable parameters managed by the administrative governance layer. This allows the system to adapt to different sensor qualities, research calibrations, and economic balancing requirements without modifying core logic.

The engine aggregates module-level eligibility signals into a single global eligibility representation for the user. This aggregation is not a simple sum but a weighted composition where each module’s contribution is scaled according to its configured importance in the system. The result of this aggregation is then passed through a smoothing function that prevents abrupt transitions between eligible and ineligible states. This smoothing is critical because biological signals naturally fluctuate, and without it, users would experience constant oscillation at the boundary of eligibility, which would destabilize both reward distribution and downstream behavioral interpretation systems.

The Eligibility Engine also incorporates temporal consistency into its evaluation model. A single high-quality measurement is not sufficient to guarantee stable eligibility if it is not supported by consistent behavior over a short historical window. For this reason, the engine maintains a lightweight rolling state of recent evaluations and uses it to stabilize eligibility transitions. This introduces a controlled inertia into the system, ensuring that eligibility reflects sustained physiological behavior rather than isolated spikes or transient artifacts.

In addition to module-level thresholds, the engine applies a global system threshold that defines the minimum overall physiological coherence required for participation in the reward pool. This global threshold acts as a baseline gate and ensures that even if individual modules temporarily exceed their local thresholds, the system does not admit users whose overall state is not sufficiently stable or meaningful. This dual-layer threshold design allows for fine-grained control of individual signal domains while preserving global consistency.

The output of the Eligibility Engine is a structured state object that contains both a binary inclusion indicator and a continuous eligibility strength coefficient. The binary indicator determines whether a user is part of the active reward pool, while the continuous coefficient influences the magnitude of their effective contribution weight inside that pool. This separation ensures that inclusion and influence are not conflated, allowing the system to model partial participation in a mathematically stable way.

Eligibility states are recalculated continuously as new biometric data arrives, but the system enforces internal stabilization rules that prevent excessive sensitivity to high-frequency fluctuations. This is achieved through controlled decay functions and rolling aggregation windows that smooth short-term noise while preserving meaningful behavioral transitions over longer periods.

The Eligibility Engine is deliberately designed to remain independent from economic logic. It does not know about rewards, token distribution, or blockchain settlement. Its only responsibility is to evaluate physiological validity in a structured and deterministic way. By isolating eligibility from economic computation, the system ensures that changes in reward mechanics do not affect the biological interpretation layer, and vice versa.

In summary, the Threshold & Eligibility Engine serves as a stabilizing gate that converts continuous, multi-dimensional biological outputs into a controlled participation signal. It ensures that only sufficiently coherent and stable physiological states contribute to the economic system, while still allowing graded participation that reflects the natural variability of human biological regulation.


## Hashrate Aggregation Engine (Unified Scalar Model — Technical Specification)

The Hashrate Aggregation Engine is the central normalization layer that converts multi-dimensional, module-specific biological contributions into a single scalar value representing a user’s economic influence inside the Weighted Continuous Selection Pool. This layer is the first point in the system where heterogeneous physiological signals become fully unified into a single computation space that is used for reward distribution.

The engine operates only on already processed and normalized inputs. It does not access raw biometric signals and does not perform any form of signal interpretation. Its only responsibility is to take the outputs of the Module System, apply system-wide weighting rules, incorporate eligibility constraints, and produce a deterministic scalar value that represents the user’s current hashrate.

Each user at any given moment has a set of active module outputs. Each module output consists of a normalized contribution value and an associated confidence score. These values are the result of upstream processing and already include baseline normalization relative to the user’s long-term physiological profile. The Aggregation Engine treats these module outputs as atomic inputs and does not attempt to reinterpret or decompose them.

The core computation of the engine is a weighted composition of all active module contributions. Each module is assigned a configurable weight that defines its relative importance in the overall hashrate computation. These weights are not embedded in the module implementation itself but are defined externally through the administrative configuration system. This separation ensures that module logic remains stable while system-level economic priorities can be adjusted dynamically.

In addition to module weights, the engine incorporates confidence values into the aggregation process. Confidence acts as a multiplicative stabilizer that reduces the influence of unreliable or noisy measurements without discarding them entirely. This ensures that low-quality signals contribute less to the final hashrate but are not abruptly excluded, preserving continuity in the system and preventing instability caused by binary filtering at the aggregation stage.

The aggregation process also integrates the output of the Eligibility Engine. Eligibility does not directly modify module values but influences the final hashrate through a global participation coefficient. This coefficient represents how strongly the user is currently admitted into the reward pool and scales the final aggregated result accordingly. Users with partial eligibility contribute proportionally reduced influence, while fully eligible users contribute at full weight.

The resulting computation produces a single scalar value that represents the user’s effective hashrate at a given moment. This value is always non-negative and continuously varying over time. It reflects not only the instantaneous physiological state of the user but also the structural configuration of their biometric modules and their current eligibility status within the system.

The engine is designed to be fully deterministic and replayable. Given the same sequence of module outputs, confidence values, weights, and eligibility states, the Aggregation Engine must always produce identical hashrate results. This requirement is critical for downstream systems such as fraud detection, reward settlement, and blockchain reconciliation, all of which depend on reproducibility of economic state.

The Aggregation Engine also supports incremental computation. Instead of recalculating the full hashrate from scratch on every update, the system maintains a rolling state of active module contributions and applies delta updates as new biometric data arrives. This allows the system to operate efficiently in high-frequency environments without sacrificing correctness or determinism.

Although the engine produces a single scalar output, it internally maintains structured traceability data for each computation step. This includes per-module contribution breakdowns, applied weights, confidence adjustments, and eligibility scaling factors. This traceability is not used in real-time economic computation but is essential for debugging, auditing, and fraud analysis.

The Aggregation Engine is intentionally decoupled from reward logic. It does not compute reward shares, does not interact with emission schedules, and does not perform any form of financial interpretation. Its sole output is a normalized biological influence scalar that downstream systems use as the basis for economic distribution.

In summary, the Hashrate Aggregation Engine transforms heterogeneous, confidence-weighted, and eligibility-filtered biometric module outputs into a single deterministic scalar value. This value represents the unified expression of a user’s physiological contribution to the system and serves as the primary input for all economic reward computations.


## Weighted Continuous Selection Pool (Core Economic Model — Technical Specification)

The Weighted Continuous Selection Pool is the central economic mechanism of the system. It defines how reward value is distributed among participants based on continuously evolving biological state rather than discrete time-based events or probabilistic selection. This layer is the first point where computed hashrate values are transformed into economic meaning.

The system maintains a continuously updated set of active participants. A participant enters this pool only when the Eligibility Engine determines that their current physiological state satisfies the minimum system-wide participation constraints. Pool membership is therefore dynamic and state-dependent rather than static or identity-based. A user is not permanently part of the reward system; they are part of it only while their biological state qualifies.

Once inside the pool, each participant is assigned an effective weight. This weight is derived from the output of the Hashrate Aggregation Engine and represents the user’s relative economic influence at a given moment. The effective weight is not a raw hashrate value. It is a transformed value that incorporates eligibility strength, trust modifiers, and confidence scaling, producing a normalized contribution metric that is suitable for proportional distribution.

The pool itself is a continuously evolving normalization space. At any given moment, the system maintains the sum of all effective weights of currently eligible users. This sum defines the normalization denominator for reward allocation. Each participant’s share of the reward is computed as the ratio of their individual effective weight to the total weight of all participants in the pool. This ensures that reward distribution remains strictly proportional and fully deterministic.

The economic model explicitly avoids any form of random selection or lottery-based allocation. Although biological inputs may fluctuate and produce variability in individual user weights, the system itself does not introduce stochasticity. Any variability in outcomes is a direct consequence of changes in physiological state rather than probabilistic decision-making within the reward system.

The pool operates continuously in real time, meaning that user weights and eligibility states are updated as new biometric data is processed. However, for purposes of persistence, settlement, and blockchain interaction, the continuous state is periodically sampled into deterministic snapshots. These snapshots represent consistent views of the pool at specific moments in time and are used as the basis for downstream reward settlement processes.

Transitions into and out of the pool are handled as gradual state changes rather than abrupt binary switches. When a user approaches the eligibility boundary, their effective contribution is smoothly scaled rather than instantly included or excluded. This prevents oscillatory behavior where users repeatedly enter and exit the pool due to small fluctuations in physiological measurements. Instead, the system models participation as a continuous function of eligibility strength.

The Weighted Continuous Selection Pool is designed to support high-frequency updates without requiring global recomputation. Each user update affects only their local contribution to the pool, and the system maintains an incremental aggregation of total weight. This allows the system to scale to large numbers of concurrent participants without degrading performance or requiring full recomputation of the global state on every update.

The pool is independent of reward timing logic. It does not define when rewards are emitted or settled. It only defines how rewards would be distributed if a settlement event occurs at a given moment. This separation ensures that the economic model remains continuous and state-driven, while settlement layers can operate on independent batching schedules without affecting the correctness of the underlying distribution logic.

The Weighted Continuous Selection Pool therefore serves as the real-time economic substrate of the system. It continuously maps physiological state into proportional economic influence, ensuring that reward distribution is always aligned with current biological contribution strength rather than historical averages or arbitrary time windows.


## Admin & Governance System (Technical Specification)

The Admin & Governance System defines the control plane of the entire Bio-Proof-of-Brain economic protocol. It is responsible for configuring, evolving, and constraining all non-deterministic parameters of the system without ever interfering with the deterministic execution layers such as signal processing, hashrate computation, eligibility evaluation, or reward settlement logic. The fundamental principle of this layer is that it can change the shape of the system, but it cannot influence individual runtime computations directly.

The system is designed as a strictly separated configuration authority that produces versioned, auditable configuration states consumed by all runtime services. These configuration states are immutable once published, meaning that any change to system behavior must occur through the creation of a new configuration version rather than in-place mutation. This ensures full reproducibility of historical system states and allows any past computation to be replayed exactly as it occurred under the configuration active at that time.

Administrative access is structured through a role-based governance model. The system defines multiple tiers of authority, each with clearly bounded capabilities. At the highest level, SuperAdmin roles have the ability to define system structure, including enabling or disabling biometric modules, modifying module schemas, and adjusting global economic policies such as reward scaling curves and emission parameters. SuperAdmins are the only role permitted to introduce structural changes that affect the composition of the system itself.

Admin-level roles operate within the constraints defined by SuperAdmins and are responsible for operational tuning of the system. This includes adjusting module weights, modifying threshold parameters for eligibility, tuning sensitivity of fraud detection systems, and managing real-time economic balancing parameters. Admins do not have the ability to alter system structure or introduce new modules, but they can significantly influence the behavior of existing components through configuration changes.

Operator roles are intended for system stability management and monitoring. Operators have access to runtime controls such as pausing specific modules, disabling reward distribution temporarily, or triggering emergency fallback modes in response to anomalies or system instability. Operators do not modify economic parameters and cannot change configuration logic; their authority is strictly limited to system safety and continuity operations.

Viewer or Analyst roles exist purely for observational access. These roles are granted read-only visibility into system state, including aggregated hashrate distributions, eligibility statistics, fraud detection outputs, and historical reward flows. They are explicitly prohibited from performing any state mutation or configuration changes.

All administrative actions are executed through a versioned configuration system. Each configuration state is represented as a fully self-contained object that includes module definitions, weight distributions, threshold settings, reward policies, and risk control parameters. When a configuration is published, it is assigned a unique version identifier and becomes immediately applicable to all subsequent computations. Previous configurations remain preserved for historical replay and auditability, ensuring that system behavior can always be reconstructed exactly as it existed at any point in time.

Configuration changes are applied atomically. This means that partial updates are not allowed; either a full configuration version is activated or it is rejected. This prevents inconsistent intermediate states where different parts of the system operate under mismatched parameter sets. Atomic activation ensures that all runtime engines observe a consistent configuration boundary.

The governance system also defines explicit constraints on parameter ranges to prevent destabilizing configurations. For example, module weights must remain within normalized bounds, threshold values must maintain logical ordering relationships, and reward scaling factors must preserve non-negative economic outputs. These constraints are enforced at the configuration validation layer before any version is accepted into the active system.

In addition to static configuration control, the Admin & Governance System supports policy evolution over time. This includes the ability to introduce new modules, deprecate existing ones, and adjust economic interpretation rules as the system matures. However, all such changes must be introduced through explicit version transitions, ensuring that no silent or implicit changes affect system behavior.

A critical aspect of the governance system is auditability. Every configuration change is recorded in an append-only audit log that includes the identity of the actor, the exact configuration diff, the resulting version hash, and the timestamp of activation. This audit log is cryptographically verifiable and serves as the canonical record of system evolution.

The Admin & Governance System ultimately functions as the control surface of the protocol, allowing structured evolution of the biological mining economy while preserving strict determinism in all runtime computations. It ensures that while the system can adapt, it can never behave unpredictably or inconsistently across historical states.


## User State System — Database Layer (Technical Specification)

The User State System defines the persistent storage layer responsible for maintaining all runtime-relevant representations of a user’s biological, economic, and behavioral state. It is the only layer in the system that holds continuously evolving per-user state across time, bridging real-time computation engines with long-term economic settlement and behavioral analysis.

This layer is designed as a strictly append-safe and version-aware state store. It does not act as a source of computation, but rather as a canonical persistence surface for already-computed results produced by upstream engines such as the Module System, Eligibility Engine, Hashrate Aggregation Engine, and Reputation Layer. The database never interprets raw signals and never performs any form of inference. Its responsibility is to store, retrieve, and maintain historical consistency of derived state.

At the core of this system is the concept of a User Bio State. This is a continuously updated snapshot that represents the current physiological and economic status of a user as interpreted by the system. The Bio State includes the user’s current hashrate, eligibility strength, trust modifiers, and confidence levels. It is not computed inside the database layer; instead, it is written into the database as a result of upstream engine execution and serves as the authoritative reference for all downstream systems that require real-time or near-real-time user state access.

Alongside the Bio State, the system maintains a structured representation of per-module state. Each module produces its own independent state record for a user, containing normalized output values, confidence scores, and threshold evaluation results. These module states are stored separately but are always logically linked to the parent user state snapshot. This separation allows the system to reconstruct detailed historical behavior of individual physiological domains without losing alignment with global economic state.

The User State System is explicitly versioned. Every update to a user’s state is written as a new immutable record rather than modifying existing entries. This design ensures full temporal reproducibility, allowing any past state of the system to be reconstructed exactly as it existed at the time of computation. Versioning also plays a critical role in fraud detection and auditability, as it allows inconsistencies between historical states and recomputed states to be detected deterministically.

Each stored user state contains a reference to the configuration version under which it was computed. This ensures that all derived values can be traced back to the exact system parameters that produced them, including module weights, threshold settings, and eligibility rules. Without this linkage, historical replay would be impossible, as changes in configuration would otherwise invalidate prior state interpretations.

The database schema is structured to optimize both write-heavy ingestion from real-time engines and read-heavy access patterns required by reward settlement and analytics systems. Writes are append-only and optimized for high-frequency ingestion, while reads are typically performed through indexed queries over the latest state per user or over time-bounded state windows for replay and analysis.

The system maintains a separation between current state and historical state through logical views rather than physical mutation. The “current state” of a user is always derived as the latest versioned record in the system, while historical states remain fully accessible for replay, audit, and machine learning analysis. This ensures that no information is ever lost or overwritten, preserving the full evolution of user behavior over time.

In addition to real-time state storage, the User State System also maintains derived aggregates that support higher-level system components. These include rolling averages of hashrate, stability metrics over defined time windows, and compressed behavioral summaries used by the Reputation and Coaching layers. These aggregates are themselves treated as derived state and follow the same immutability and versioning rules as raw state entries.

The database layer also plays a critical role in settlement correctness. The Reward Settlement Engine relies on consistent snapshots of user state to compute deterministic reward distributions. To support this, the system guarantees snapshot integrity by ensuring that all state writes within a given computation cycle are either fully committed or not visible to downstream systems. This prevents partial updates from affecting reward calculations.

The User State System is not responsible for enforcing business logic or validation rules. It assumes that all incoming data has already been validated by upstream services. Its role is purely structural: to maintain a reliable, consistent, and fully reproducible record of system state across time.

In essence, this layer serves as the canonical memory of the entire protocol. It transforms ephemeral real-time biological computation into a persistent, queryable, and auditable state history that enables reward distribution, system analysis, fraud detection, and long-term behavioral modeling.


## Real-time Computation Layer (Technical Specification)

The Real-time Computation Layer is the execution core of the entire system where raw incoming biometric streams are continuously transformed into structured, economically meaningful state. This layer operates as a streaming processing system with strict determinism requirements, responsible for maintaining the live bridge between biological input and economic output.

Its primary role is not storage, not settlement, and not governance. It is the active computation surface where physiological signals are interpreted, normalized, evaluated, and continuously converted into intermediate representations that downstream systems consume. Everything in this layer is ephemeral by design, except for the structured outputs that are explicitly forwarded to persistence and aggregation layers.

The system operates as a continuous event-driven pipeline. Incoming biometric signals arrive as time-indexed streams originating from multiple device sources and classification modules. These streams are not processed in isolation. Instead, they are merged into a unified per-user computation context that represents the current biological state of the user across all active modalities. This context is continuously updated as new signal frames arrive.

Each update triggers a recomputation cycle within a bounded local window of time. This window is not a fixed business epoch but a sliding computational buffer that holds the most recent state of each module. Within this buffer, the system maintains the latest normalized values, confidence scores, and stability indicators for all active biometric channels. This ensures that the system always operates on the most recent and coherent representation of user physiology rather than on isolated signal samples.

Within each computation cycle, module-level processing is performed first. Each active module independently updates its internal state based on incoming signal deltas. These updates are deterministic and stateless with respect to external systems, meaning that given identical input streams, a module will always produce identical outputs. Module computation is isolated to ensure that failures or inconsistencies in one modality do not propagate to others.

Once module-level updates are complete, the system transitions into state synthesis. In this phase, all module outputs are combined into a unified per-user biological state object. This object contains normalized module contributions, confidence-weighted adjustments, and eligibility-relevant indicators. It represents the canonical real-time snapshot of the user’s physiological condition as interpreted by the system.

The synthesized state is then passed to the Hashrate Aggregation Engine, which computes the user’s current economic influence scalar. This computation is performed in real time for every relevant update cycle, ensuring that the system always maintains an up-to-date representation of each user’s contribution strength. However, this value is not immediately persisted as a final economic record. Instead, it is stored in an intermediate state buffer that feeds both the Eligibility Engine and the Pool Model.

The Eligibility Engine is invoked as part of the same computation cycle. It evaluates whether the updated state qualifies the user for participation in the Weighted Continuous Selection Pool and computes a continuous eligibility coefficient. This coefficient is immediately applied to the user’s effective weight, ensuring that inclusion and influence remain synchronized at all times.

The Real-time Computation Layer maintains strict ordering guarantees within each user context. Updates for a single user are processed sequentially to prevent race conditions between overlapping signal updates. However, the system is horizontally scalable across users, meaning that different user contexts can be processed in parallel without shared-state contention.

The layer also maintains a lightweight in-memory representation of active user states. This in-memory state is not authoritative and is treated as a cache of the latest computed values. The authoritative state is always persisted in the User State System, but real-time decisions are made using this in-memory representation to ensure low-latency computation.

A critical property of this layer is bounded latency. All computation cycles must complete within a strict time window to ensure that system responsiveness remains consistent even under high load. If processing exceeds this bound, the system applies controlled degradation strategies such as reducing update frequency, prioritizing critical modules, or temporarily lowering confidence weighting for non-essential signals.

The Real-time Computation Layer also serves as the primary integration point for fraud detection and quality assessment signals. However, these systems do not block computation. Instead, they inject modifiers into module confidence and eligibility calculations. This ensures that even under suspected anomaly conditions, the system remains operational while gradually adjusting trust in affected signals.

At the end of each computation cycle, the resulting state is emitted to downstream systems in three forms. The first is a transient in-memory update used for immediate reward pool calculation. The second is a persisted snapshot written to the User State System. The third is a structured event emitted to the Reward Settlement pipeline for later aggregation and blockchain settlement.

The Real-time Computation Layer is therefore the central nervous system of the entire architecture. It continuously transforms raw biological input into structured economic meaning while maintaining strict determinism, horizontal scalability, and low-latency execution constraints.


## Reward Settlement Layer (Technical Specification)

The Reward Settlement Layer is the final deterministic bridge between the continuously evolving biological economy and the irreversible external financial system. Its responsibility is not to compute value, not to interpret physiological state, and not to influence reward logic. Its sole function is to take already-finalized economic state snapshots produced by the Real-time Computation Layer and transform them into auditable, batchable, and blockchain-compatible settlement instructions.

This layer exists because the internal system operates as a continuous state machine, while the blockchain operates as a discrete transaction system. The Reward Settlement Layer resolves this mismatch by converting continuous weighted participation data into structured settlement batches that can be safely executed on-chain without losing determinism or introducing inconsistencies.

The input to this layer is a sequence of fully resolved Reward Pool Snapshots. Each snapshot contains a complete representation of the Weighted Continuous Selection Pool at a specific moment in time, including the set of eligible users, their effective weights, their confidence-adjusted hashrates, and all associated system modifiers such as trust and eligibility coefficients. These snapshots are immutable once created and represent the canonical economic truth for their respective computation window.

The Reward Settlement Layer does not modify these snapshots in any way. It operates strictly as a transformation and aggregation system. Its first responsibility is to group multiple snapshots into settlement batches based on system-defined batching policies. These policies are not time-based in the abstract sense, but rather based on constraints such as maximum batch size, maximum gas cost exposure, and minimum stability thresholds of pool composition.

Once a batch is formed, the system computes deterministic reward allocations for each user across the entire batch. This computation involves aggregating each user’s effective weight contributions across all snapshots included in the batch and normalizing them against the total aggregated pool weight. The result is a precise, deterministic mapping from biological contribution space into economic reward space.

The output of this computation is a structured settlement object that contains per-user reward deltas. These deltas represent the exact amount of reward each user is entitled to claim for the given batch window. No randomness is introduced at any stage of this process, and identical input batches must always produce identical output settlement objects.

Before any settlement object is accepted for blockchain submission, the layer performs a full consistency verification pass. This verification ensures that the sum of all user allocations exactly matches the total reward emission defined for the batch window, that no user is double-counted across overlapping batches, and that all input snapshots originate from valid and finalized system states. If any inconsistency is detected, the entire batch is rejected and flagged for replay by upstream systems. The Reward Settlement Layer is not allowed to repair or adjust inconsistent state; it can only accept or reject.

Once a settlement batch passes validation, it is transformed into blockchain-compatible execution instructions. These instructions are structured in a way that minimizes on-chain complexity and gas consumption while preserving full auditability. Depending on system configuration and scale, this may take the form of direct batched state updates, Merkle root commitments, or hybrid models where partial computation is performed off-chain and verification is performed on-chain.

The layer also maintains strict idempotency guarantees. Each settlement batch is assigned a unique deterministic identifier derived from its input snapshot hashes. This ensures that the same batch can never be processed twice, even in the presence of retries, network failures, or partial system outages. Idempotency is a critical property because it guarantees that financial state cannot diverge due to operational instability.

All settlement operations are recorded in an append-only Settlement Ledger. This ledger contains full traceability information for each batch, including input snapshot references, computed allocation hashes, final transaction payloads, blockchain transaction identifiers, and execution outcomes. This ledger serves as the ultimate audit layer for the entire economic system and is designed to support full historical reconstruction of financial state.

The Reward Settlement Layer operates in close coordination with the Oracle Settlement Layer, which executes the final on-chain submission. However, the Settlement Layer itself does not interact with private keys, signing infrastructure, or blockchain communication channels. Its responsibility ends at the generation of cryptographically consistent and verifiable transaction payloads.

The system is designed to tolerate partial failures at the settlement boundary without affecting internal state consistency. If a blockchain submission fails, the corresponding settlement batch remains valid and can be retried without recomputation, as long as the input snapshot set remains unchanged. If input state changes, a new batch must be generated, preserving strict separation between computation and execution.

In essence, the Reward Settlement Layer acts as the deterministic financial compiler of the entire system. It takes continuous biological participation data, compresses it into structured economic obligations, and produces final execution-ready representations that can be safely committed to an external decentralized ledger without ambiguity, randomness, or loss of traceability.


## Anti-Fraud Integration Point (Technical Specification)

The Anti-Fraud Integration Point is a transversal control layer embedded across the entire system rather than a standalone processing stage. Its role is to continuously evaluate the integrity of biometric signals, module outputs, and derived economic state without ever becoming a blocking dependency for real-time computation. The guiding principle of this layer is that fraud detection must be non-invasive to system flow while still capable of progressively reducing the economic influence of suspicious behavior.

This layer operates on the assumption that fraud cannot be eliminated at the signal level, especially in a system that relies on consumer-grade or semi-trusted biometric hardware. Instead of attempting to enforce absolute correctness, the system models fraud as a probabilistic degradation of trust over time. Every user, signal source, and module output exists within a continuously updated trust landscape that evolves based on behavioral consistency, statistical deviation, and cross-modal correlation analysis.

The Anti-Fraud Integration Point receives inputs from all upstream systems, including raw module outputs, aggregated hashrate values, eligibility states, and historical behavioral trajectories stored in the User State System. It does not introduce new primary computations of biological meaning. Instead, it evaluates consistency between expected physiological patterns and observed system behavior across time. Any deviation from expected structural coherence results in a reduction of confidence and trust coefficients that propagate downward into the Hashrate Aggregation Engine and Eligibility Engine.

Fraud evaluation is performed at multiple independent levels. At the signal level, the system analyzes the internal consistency of individual biometric streams, checking for artifacts such as signal flatlining, unnatural periodicity, or physically implausible transitions. At the module level, it evaluates whether derived outputs remain consistent with known physiological constraints and with the user’s historical baseline profile. At the cross-module level, it checks whether different physiological domains exhibit realistic correlations, such as expected coupling between heart rate variability and respiratory patterns or between EEG state shifts and relaxation indices.

The output of the Anti-Fraud Integration Point is not a binary fraud flag. Instead, it produces a multi-dimensional trust vector that influences downstream systems through continuous weighting adjustments. Each user, each module, and each signal source has an associated trust coefficient that evolves over time. These coefficients directly modulate confidence values in the Hashrate Aggregation Engine and eligibility strength in the Eligibility Engine, effectively reducing economic influence without explicitly excluding the user unless extreme degradation thresholds are reached.

In addition to continuous trust scoring, the system maintains anomaly event detection for significant deviations. These events are not used for immediate enforcement but are recorded in a fraud event ledger for later aggregation, pattern analysis, and potential administrative review. This allows the system to distinguish between transient anomalies, such as sensor noise or environmental interference, and persistent fraudulent behavior patterns that indicate systematic manipulation.

A critical design constraint of this layer is that it must never block real-time computation. Even in cases of severe anomaly detection, the system continues to process signals and compute hashrate values. The Anti-Fraud Integration Point only modifies the influence of those values rather than halting their existence. This ensures that system liveness is preserved under all conditions, which is essential for maintaining continuity in both user experience and economic state tracking.

The trust model used by this layer is inherently temporal. Short-term anomalies have limited impact unless they are reinforced by repeated patterns over time. Conversely, long-term consistency gradually increases trust, allowing users with stable physiological behavior to gain stronger economic weighting. This creates a dynamic where the system rewards not only high-quality biological signals but also sustained behavioral reliability.

Cross-user statistical analysis is also incorporated into the fraud detection model. The system maintains population-level distributions of biometric features and continuously evaluates whether individual users deviate significantly from expected ranges. However, such deviations are not automatically considered fraudulent, as the system explicitly accounts for biological diversity. Instead, deviation contributes only to probabilistic anomaly scoring, which must be reinforced by additional signals before influencing trust reduction.

The Anti-Fraud Integration Point is tightly coupled with the configuration system, allowing administrative control over sensitivity thresholds, anomaly detection sensitivity, and trust decay rates. These parameters are versioned and subject to the same governance rules as all other system configurations, ensuring that fraud detection behavior remains reproducible and auditable over time.

All outputs of this layer are ultimately expressed as modifiers applied to existing system variables rather than standalone decisions. This includes adjustments to module confidence scores, eligibility strength coefficients, and effective hashrate scaling factors. By embedding fraud detection as a modulation layer rather than a decision layer, the system ensures that economic continuity is preserved while still progressively isolating and neutralizing suspicious behavior patterns.

In essence, the Anti-Fraud Integration Point functions as a continuous trust shaping mechanism that operates across all levels of the system. It does not decide who is valid or invalid in a binary sense; instead, it continuously reshapes the economic influence landscape based on observed behavioral integrity, ensuring that the system remains robust against manipulation while preserving fluid participation dynamics.
